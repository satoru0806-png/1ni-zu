# Requirements: life-navigator

## 1. 目的（Executive Summary）
Life Navigatorは、1名のユーザーが人生の4領域（人間関係・お金・仕事・健康）を毎日記録・振り返り・可視化するWebアプリケーションである。習慣化により自己改善サイクルを支援する。

## 2. 背景 & Context
- ユーザーは日々の目標設定・振り返りを継続する仕組みを持っていない。
- 紙のジャーナルやメモアプリでは、構造化されたデータ蓄積と傾向可視化ができない。
- 既存の習慣トラッカーは汎用的すぎ、4領域に特化したスコアリング機能がない。

## 3. スコープ
### 3.1 In Scope
- 夢の登録（最大3件）
- 毎日のMIT（Most Important Tasks）3件の登録
- 夜の振り返り（できたこと・感謝・明日やること）の記録
- 4領域スコア（人間関係・お金・仕事・健康）の0〜100での表示・編集
- テキストメモの入力、ルールベース要約（120文字以内）、タスク抽出
- 過去7日分の履歴一覧と日別詳細表示
- スマートウォッチ向けミニビュー（MIT表示・スコア表示）
- ローカル環境での動作（localhost）

### 3.2 Out of Scope（重要）
- ユーザー認証・マルチユーザー対応
- LLM APIを使用したAI処理（MVP後に対応）
- 音声入力・音声認識
- プッシュ通知・リマインダー
- クラウドデプロイ・ホスティング
- データのエクスポート・インポート
- モバイルネイティブアプリ

## 4. 用語集 / Glossary
| 用語 | 定義 |
|------|------|
| MIT | Most Important Tasks。その日に必ず達成すべき3つのタスク。 |
| DayLog | 1日分の全データ（MIT、スコア、振り返り、メモ）を格納する1レコード。dateフィールド（YYYY-MM-DD）で一意に識別される。 |
| Dream | ユーザーが登録する夢・目標。最大3件まで。 |
| 4領域スコア | 人間関係（relationshipScore）、お金（moneyScore）、仕事（workScore）、健康（healthScore）の4項目を0〜100の整数で評価した値。 |
| 振り返り | 1日の終わりに記録する「できたこと（doneNote）」「感謝（gratitudeNote）」「明日やること（tomorrowPlan）」の3項目。 |
| メモ処理 | テキストメモ（memoRaw）から要約（memoSummary）とタスク一覧（memoTasksJson）を生成する処理。 |
| ウォッチビュー | スマートウォッチ画面サイズ（194x194px円形）に合わせた閲覧専用ビュー。 |

## 5. ステークホルダー & 役割
| Role | 権限/責務 |
|------|----------|
| エンドユーザー | アプリの全機能を使用する唯一のユーザー。データの入力・閲覧・編集を行う。 |
| 開発者 | アプリの開発・保守を行う。ローカル環境でのセットアップ・起動を実施する。 |

## 6. 前提/仮定（Assumptions）
- A-001: ユーザーは1名のみ。認証機能は不要とする。
- A-002: データはローカルのSQLiteファイルに保存され、バックアップはユーザーの責任とする。
- A-003: AI処理はMVPではルールベースで実装し、将来LLM API（OpenAI/Claude）に差し替え可能な非同期関数インターフェースを維持する。
- A-004: ブラウザはChrome/Edge/Safariの最新版を対象とし、IE対応は行わない。
- A-005: タイムゾーンはユーザーのブラウザのローカルタイムゾーンに従う。日付はYYYY-MM-DD文字列で管理する。

## 7. 制約（Constraints）
- C-001: フロントエンドはNext.js（App Router）、TypeScript、Tailwind CSSで実装する。
- C-002: データベースはSQLite、ORMはPrismaを使用する。
- C-003: 実行環境はNode.js 18以上とする。
- C-004: 外部APIへの依存はMVPでは0件とする（全処理をローカルで完結させる）。
- C-005: データモデルはDreamテーブルとDayLogテーブルの2テーブル構成とする。

## 8. 成功条件（Success Metrics）
| Metric | Target | Measurement Method |
|--------|--------|-------------------|
| 全5画面のHTTP応答 | 5/5画面でHTTP 200を返す | next buildが成功し、各ルートにアクセスして200応答を確認する |
| データの保存と読み出し | CRUD操作で保存データと取得データが一致する | 各APIエンドポイントにPOSTしたデータとGETで取得したデータを比較する |
| メモからのタスク抽出 | 動詞パターンを含む行がタスクとして抽出される | 動詞を含むメモを保存し、memoTasksJsonに該当行が含まれることを確認する |
| ビルド成功 | next buildの終了コードが0 | ローカルでnext buildを実行する |

## 9. 機能要件（Functional Requirements）

### REQ-001: 夢の登録
- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーが夢登録フォームにテキストを入力し送信ボタンを押したとき、システムはそのテキストをDreamテーブルに1件追加し画面に表示しなければならない。
- 根拠/目的: ユーザーが自分のビジョンを明確にし、毎日の行動指針とするため。
- 受入テスト(GWT):
  - AT-001: Given Dreamテーブルに0件のデータがある When ユーザーが「世界一周する」と入力し送信ボタンを押す Then Dreamテーブルに1件のレコードが追加されtextフィールドが「世界一周する」である
  - AT-002: Given Dreamテーブルに3件のデータがある When ユーザーが新しい夢を登録しようとする Then 入力フォームが非表示であり登録操作が不可能である
- 例外・エラー:
  - EH-001: ユーザーが空文字列で送信した場合、システムはリクエストを無視しデータベースに保存しないこと。
  - EH-002: Dreamテーブルに既に3件存在する状態でPOSTリクエストが送信された場合、システムはHTTP 400を返しデータベースに保存しないこと。

### REQ-002: 夢の削除
- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーが夢の削除ボタンを押したとき、システムは該当DreamレコードをDreamテーブルから削除し画面から除去しなければならない。
- 根拠/目的: ユーザーが夢を変更・整理できるようにするため。
- 受入テスト(GWT):
  - AT-003: Given Dreamテーブルにid=1のレコードがある When ユーザーがid=1の削除ボタンを押す Then Dreamテーブルからid=1のレコードが削除され画面に表示されない
- 例外・エラー:
  - EH-003: 存在しないidの削除リクエストが送信された場合、システムはエラーレスポンスを返しHTTP 500を返さないこと。

### REQ-003: MITの登録と更新
- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーがtodayページでMIT3件を入力し保存ボタンを押したとき、システムは当日日付のDayLogレコードをupsertしmit1とmit2とmit3フィールドに保存しなければならない。
- 根拠/目的: ユーザーがその日の最重要タスク3件を明確にするため。
- 受入テスト(GWT):
  - AT-004: Given 当日のDayLogレコードが存在しない When ユーザーが「報告書作成」「会議準備」「運動30分」と入力し保存する Then DayLogテーブルに当日日付のレコードが1件作成されmit1が「報告書作成」でmit2が「会議準備」でmit3が「運動30分」である
  - AT-005: Given 当日のDayLogレコードが既に存在する When ユーザーがmit1を「企画書作成」に変更して保存する Then 既存レコードのmit1が「企画書作成」に更新されレコード数は増加しない
- 例外・エラー:
  - EH-004: MIT3件すべてが空文字列の場合、システムはnull値でDayLogレコードを保存すること。

### REQ-004: 4領域スコアの登録と更新
- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーがtodayページでスライダーを操作して保存ボタンを押したとき、システムは当日日付のDayLogレコードの4スコアフィールドを0以上100以下の整数で保存しなければならない。
- 根拠/目的: ユーザーが人生の4領域の状態を定量的に把握し変化を追跡するため。
- 受入テスト(GWT):
  - AT-006: Given 当日のDayLogレコードが存在しない When ユーザーがrelationshipScore=80とmoneyScore=60とworkScore=70とhealthScore=90に設定し保存する Then DayLogテーブルに当日日付のレコードが作成され各スコアが指定値と一致する
  - AT-007: Given 前日のDayLogレコードにrelationshipScore=75が保存されている When ユーザーが当日のtodayページを開く Then relationshipScoreの初期値が75である
- 例外・エラー:
  - EH-005: スコアが0未満または100超の値でAPIに送信された場合、システムは0から100の範囲にクランプして保存すること。

### REQ-005: 夜の振り返りの登録と更新
- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーがreviewページで3項目を入力し保存ボタンを押したとき、システムは当日日付のDayLogレコードのdoneNoteとgratitudeNoteとtomorrowPlanをupsertで保存しなければならない。
- 根拠/目的: ユーザーが毎日の振り返り習慣を形成しポジティブな自己認識を促進するため。
- 受入テスト(GWT):
  - AT-008: Given 当日のDayLogレコードが存在する When ユーザーがdoneNote=「プレゼン完了」とgratitudeNote=「チームのサポート」とtomorrowPlan=「資料修正」と入力し保存する Then 既存レコードの該当フィールドが更新されている
- 例外・エラー:
  - EH-006: 3項目すべてが空文字列で保存された場合、システムはnull値でDayLogレコードを保存すること。

### REQ-006: メモの入力とタスク抽出
- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーがtodayページでメモを入力し保存ボタンを押したとき、システムはmemoRawに原文を保存しprocessMemo関数を実行してmemoSummaryとmemoTasksJsonを生成し保存しなければならない。
- 根拠/目的: ユーザーのメモから自動的にタスクを抽出し行動の明確化を支援するため。
- 受入テスト(GWT):
  - AT-009: Given 空のDayLogレコード When ユーザーが「山田さんに連絡する」と「企画書を作成する」と「牛乳を買う」を改行区切りで入力し保存する Then memoTasksJsonに3件のタスクが含まれる
  - AT-010: Given 200文字のメモテキスト When processMemo関数が実行される Then memoSummaryの文字数が123以下である
  - AT-011: Given タスク動詞を含まないメモ「今日はいい天気だった」 When processMemo関数が実行される Then memoTasksJsonが空配列である
- 例外・エラー:
  - EH-007: memoRawが空文字列の場合、システムはメモ処理を実行せずmemoSummaryとmemoTasksJsonをnullのまま保存すること。
- 補足:
  - タスク抽出対象動詞: する、やる、確認、連絡、買う、送る、調べる、作る、書く、読む、行く、始める、終わる、提出、報告、準備、予約、申し込み
  - 関連: REQ-904

### REQ-007: ダッシュボード表示
- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーがdashboardページにアクセスしたとき、システムは当日の4スコアと平均値と当日のMIT一覧と登録済みの夢一覧と過去7日分のログサマリーを表示しなければならない。
- 根拠/目的: ユーザーが現在の状態と直近の傾向を一目で把握するため。
- 受入テスト(GWT):
  - AT-012: Given 当日のDayLogにrelationshipScore=80とmoneyScore=60とworkScore=70とhealthScore=90が保存されている When ユーザーがdashboardを開く Then 総合スコアとして75が表示される
  - AT-013: Given 過去7日間にDayLogが3日分存在する When ユーザーがdashboardを開く Then 7日分の一覧が表示されデータがある3日はスコアが表示されデータがない4日は「No data」と表示される
- 例外・エラー:
  - EH-008: 当日のDayLogが存在しない場合、システムは前日のスコアを引き継いだ値を表示すること。前日もなければ50を表示すること。

### REQ-008: 履歴一覧表示
- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーがhistoryページにアクセスしたとき、システムは過去7日分のDayLogを日付降順で一覧表示しなければならない。
- 根拠/目的: ユーザーが過去の記録を振り返り傾向を把握するため。
- 受入テスト(GWT):
  - AT-014: Given 過去7日間に5日分のDayLogがある When ユーザーがhistoryを開く Then 7日分のカードが表示され5日分はデータ付きで2日分は「記録なし」と表示される
- 例外・エラー:
  - EH-009: DayLogが1件も存在しない場合、システムは7日分の「記録なし」カードを表示すること。

### REQ-009: 履歴日別詳細表示
- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーがhistoryページの一覧から特定の日付をクリックしたとき、システムはその日のDayLogの全フィールドを詳細表示しなければならない。
- 根拠/目的: ユーザーが特定の日の記録を詳しく確認するため。
- 受入テスト(GWT):
  - AT-015: Given 2026-03-05のDayLogにmit1=「報告書」とdoneNote=「完了」とmemoTasksJson=["買い物"]がある When ユーザーが2026-03-05の行をクリックする Then MIT欄に「報告書」と振り返り欄に「完了」とメモ欄に「買い物」が表示される
- 例外・エラー:
  - EH-010: データがない日付がクリックされた場合、システムはクリックイベントを無視すること。

### REQ-010: ウォッチ用ミニビュー
- 種別: EARS-イベント駆動
- 優先度: SHOULD
- 要件文(EARS): ユーザーがwatchページにアクセスしたとき、システムは194x194ピクセルの円形エリア内にMITタブとScoreタブを表示しなければならない。
- 根拠/目的: スマートウォッチ画面サイズで当日の状態を確認できるようにするため。
- 受入テスト(GWT):
  - AT-016: Given 当日のDayLogにmit1=「会議」とmit2=「資料作成」が保存されている When ユーザーがwatchを開きMITタブを選択する Then 「2/3」と「会議」と「資料作成」が表示される
  - AT-017: Given 当日のスコアが全て50 When ユーザーがwatchを開きScoreタブを選択する Then 平均値「50」と4本のバーが表示される
- 例外・エラー:
  - EH-011: DayLogが存在しない場合、システムはMITタブで「MIT未登録」を表示しScoreタブでデフォルト値50を表示すること。

### REQ-011: ページ間ナビゲーション
- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムはwatchページを除く全ページの画面下部にdashboardとtodayとreviewとhistoryとonboardingへの遷移リンクを含む固定ナビゲーションバーを表示しなければならない。
- 根拠/目的: ユーザーが全ページ間を1タップで移動でき毎日の記録フローを維持するため。
- 受入テスト(GWT):
  - AT-018: Given ユーザーがtodayページにいる When ナビバーの「Review」をクリックする Then reviewページに遷移しナビバーの「Review」がアクティブ状態で表示される
  - AT-019: Given ユーザーがwatchページにいる When 画面を確認する Then ナビゲーションバーが表示されていない
- 例外・エラー:
  - EH-012: 存在しないパスにアクセスした場合、システムはNext.jsのデフォルト404ページを表示すること。

### REQ-012: ルートリダイレクト
- 種別: EARS-イベント駆動
- 優先度: MUST
- 要件文(EARS): ユーザーがルートパスにアクセスしたとき、システムはdashboardページにHTTPリダイレクトしなければならない。
- 根拠/目的: アプリの入り口をダッシュボードに統一するため。
- 受入テスト(GWT):
  - AT-020: Given サーバーが起動している When ユーザーがルートパスにアクセスする Then dashboardにリダイレクトされダッシュボードが表示される
- 例外・エラー:
  - EH-013: サーバーが起動していない場合、ブラウザが接続エラーを表示する（システム側の対応不要）。

### REQ-013: スコアの前日引き継ぎ
- 種別: EARS-状態駆動
- 優先度: MUST
- 要件文(EARS): 当日のDayLogレコードが存在しないとき、システムは直近の過去DayLogレコードのスコア値をスコア取得APIで返却しなければならない。
- 根拠/目的: ユーザーが毎日スコアを再入力する手間を削減するため。
- 受入テスト(GWT):
  - AT-021: Given 2026-03-05のDayLogにrelationshipScore=80がありかつ2026-03-06のDayLogが存在しない When 2026-03-06にGET /api/daylogを実行する Then relationshipScore=80が返る
  - AT-022: Given DayLogテーブルにレコードが0件 When GET /api/daylogを実行する Then 4スコアすべてが50で返る
- 例外・エラー:
  - EH-014: データベース読み取りエラーが発生した場合、システムはHTTP 500を返すこと。

## 10. 非機能要件（Non-Functional Requirements）

### REQ-901: ビルド成功
- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムはnext buildコマンド実行時にTypeScriptエラー0件かつビルドエラー0件で終了コード0を出力しなければならない。
- 根拠/目的: デプロイ可能な品質を保証するため。
- 受入テスト(GWT):
  - AT-023: Given プロジェクトの全ソースコード When npx next buildを実行する Then 終了コードが0である
- 例外・エラー:
  - EH-015: ビルドエラーが発生した場合、TypeScriptまたはNext.jsのエラーメッセージが標準出力に出力されること。

### REQ-902: レスポンス時間
- 種別: EARS-普遍
- 優先度: SHOULD
- 要件文(EARS): システムはすべてのAPIエンドポイントにおいてローカル環境でリクエスト受信から500ミリ秒以内にレスポンスを送信しなければならない。
- 根拠/目的: ユーザーが操作のたびに待ち時間を感じないようにするため。
- 受入テスト(GWT):
  - AT-024: Given サーバーが起動している When GET /api/daylogを10回実行する Then 全リクエストのレスポンスタイムが500ミリ秒以内である
- 例外・エラー:
  - EH-016: DBファイルが破損しPrismaがエラーを返す場合、システムは500ミリ秒以内にHTTP 500を返すこと。

### REQ-903: データ永続性
- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムはSQLiteファイルにすべてのデータを永続化しサーバー再起動後もデータを保持しなければならない。
- 根拠/目的: ユーザーのデータが失われないことを保証するため。
- 受入テスト(GWT):
  - AT-025: Given DayLogに1件のデータがある When サーバーを停止し再起動する Then GET /api/daylogで同じデータが返る
- 例外・エラー:
  - EH-017: dev.dbファイルが削除された場合、システムはPrismaマイグレーション実行により空のデータベースを再作成すること。

### REQ-904: AI処理の差し替え可能性
- 種別: EARS-普遍
- 優先度: MUST
- 要件文(EARS): システムはメモ処理をsrc/lib/ai.tsの非同期関数として実装し戻り値の型を変更せずに内部実装を差し替え可能としなければならない。
- 根拠/目的: MVPからAI強化版への移行を最小工数で行うため。
- 受入テスト(GWT):
  - AT-026: Given src/lib/ai.tsファイル When summarizeMemo関数のシグネチャを確認する Then 戻り値の型がPromise<string>である
